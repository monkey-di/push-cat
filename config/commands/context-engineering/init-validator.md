# /utils:init-validator

Создай или проверь конфигурацию CLAUDE.md для этого проекта. Если CLAUDE.md пуст или не существует — создай. Если существует — верифицируй и предложи исправления.

## Принцип

CLAUDE.md реинжектируется в контекст при каждом обмене сообщениями. Каждая лишняя строка тратит токены многократно за сессию. Критерий включения: «Если убрать эту строку — Claude ошибётся?» Если нет — строка лишняя.

CLAUDE.md — индекс и набор ограничений, а не документация проекта.

## Что включать

- Команды сборки, тестирования, линтинга, которые Claude не угадает (нестандартные скрипты, флаги, последовательности)
- Правила стиля кода, которые ОТЛИЧАЮТСЯ от дефолтов языка
- Особенности тестирования (как запустить один тест, нужны ли env-переменные)
- Правила работы с git (ветки, коммиты, PR-конвенции)
- Инфраструктурные ограничения: доступы, серверы, способы деплоя
- Файлы и директории, которые нельзя трогать, с объяснением почему
- Архитектурные решения, которые нельзя вывести из кода (почему выбрана эта библиотека, почему так организовано)
- Подводные камни, о которых Claude не узнает из кода

В конце CLAUDE.md добавь секцию ссылок на все созданные файлы:
 ```
 ## Details
 - @.claude/rules/backend.md — архитектура, паттерны, подводные камни бекенда
 - @.claude/rules/frontend.md — архитектура, паттерны, подводные камни фронтенда
 - @.claude/rules/infra.md — серверы, SSH, деплой, troubleshooting
 ```
 Каждая ссылка — с кратким описанием содержимого. Этот блок нужен для навигации: CLAUDE.md редактируется и людьми, не только моделью.

## Что НЕ включать

- Файловую структуру проекта — Claude найдёт сам
- Стандартные конвенции языка — Claude их знает
- Описание компонентов и API — Claude прочитает код
- Generic советы («пишите чистый код», «обрабатывайте ошибки»)
- Команды, очевидные из package.json / Cargo.toml / pyproject.toml
- Длинные описания архитектуры — вынести в отдельные файлы

## Фаза 1: Исследование

Запусти несколько explore-субагентов параллельно (thoroughness: medium), каждый со своей областью:

**Агент 1 — Стек и структура:**
- Manifest-файлы (package.json, Cargo.toml, pyproject.toml, go.mod, etc.)
- README.md
- Структура проекта (монорепо / мультимодуль / один проект)
- Конфигурация форматтера (prettier, biome, ruff, black, rustfmt)

**Агент 2 — CI/CD и инфраструктура:**
- Конфигурации деплоя: Dockerfile, docker-compose.yml, .github/workflows/, .gitlab-ci.yml, Jenkinsfile, Procfile, vercel.json, netlify.toml, fly.toml, render.yaml, k8s-манифесты
- CI-конфигурация
- Серверные конфиги если есть в репозитории

**Агент 3 — Существующие конфигурации AI-инструментов:**
- Существующий CLAUDE.md и файлы в .claude/rules/
- .cursor/rules, .cursorrules
- .github/copilot-instructions.md, AGENTS.md
- .windsurfrules, .clinerules

Если проект маленький (один стек, нет CI, нет AI-конфигов) — достаточно одного агента вместо трёх.

После получения результатов — синтезируй: определи, что НЕ удалось понять из кода. Это вопросы к пользователю.

## Фаза 2: Уточнение с пользователем

Спроси пользователя только о том, что код не ответил:

- Есть ли серверы, к которым Claude должен подключаться? (SSH-адреса, sandbox-URL)
- Есть ли ветки или файлы, которые нельзя трогать?
- Какой процесс деплоя, если не очевиден из CI-конфигурации?
- Есть ли внешние зависимости, о которых код не знает? (микросервисы, отдельные репозитории)
- Есть ли правила, специфичные для команды? (стиль коммитов, ревью-процесс)

Не спрашивай о том, что очевидно из manifest-файлов или README.

## Фаза 3А: Создание (если CLAUDE.md пуст или не существует)

Создай CLAUDE.md с заголовком:

```
# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.
```

CLAUDE.md — не больше 50 строк. Если нужно больше — выноси в `.claude/rules/`. Каждый файл в `.claude/rules/` — не больше 50 строк. Если нужно больше — ты пишешь документацию, а не инструкции.

### Разбиение по файлам

Если проект содержит несколько независимых частей (frontend/backend, монорепо с пакетами, микросервисы) — создай отдельный файл в `.claude/rules/` для каждой части с `paths` в frontmatter:

```yaml
---
paths:
  - "my-backend/**"
---
```

Файл с `paths` загружается только при работе с файлами по указанным путям. Это значит: при покраске кнопки на фронтенде Claude не видит backend-архитектуру, при работе с API — не видит FSD-слои.

Типичная структура:
- `.claude/rules/backend.md` (paths: backend-dir/**)— архитектура, паттерны, подводные камни бекенда
- `.claude/rules/frontend.md` (paths: frontend-dir/**) — архитектура, паттерны, подводные камни фронтенда
- `.claude/rules/infra.md` (без paths) — серверы, SSH, деплой, troubleshooting

Для проектов с одним стеком — достаточно одного файла `rules/architecture.md` без paths.

Создавай только те файлы, для которых есть содержимое. Не создавай пустые шаблоны.

## Фаза 3Б: Верификация (если CLAUDE.md существует)

Прочитай существующий CLAUDE.md и все файлы в `.claude/rules/`.

Проверь:

**Актуальность:** упомянутые файлы, пути, команды — существуют ли они? Запусти команды сборки/тестирования из CLAUDE.md и проверь, что они работают.

 **Доступность серверов и URL — ОБЯЗАТЕЛЬНЫЙ ШАГ, НЕ ПРОПУСКАТЬ:**

Для каждого сервера или URL, упомянутого в CLAUDE.md или `.claude/rules/`:
 - HTTP/HTTPS URL → Fetch tool или `curl -sI --connect-timeout 5 {url} | head -1`  
 - SSH-адрес → `ssh -o ConnectTimeout=10 -o BatchMode=yes {host} echo OK 2>&1`
 - Домен без протокола → `ping -c 1 -W 5 {host} 2>&1 | head -2` (Linux) или `ping -n 1 -w 5000 {host} 2>&1 | head -4` (Windows)

 Если первая команда не дала результата — попробуй альтернативные методы: Fetch вместо curl, ping вместо SSH, увеличь timeout до 10 секунд. Вывод «недоступен» только после исчерпания всех методов.

 Если недоступен — отметь в результатах верификации как проблему с указанием, какие методы были испробованы.

**Избыточность:** есть ли информация, которую Claude найдёт сам? (файловая структура, стандартные команды, generic советы, описания компонентов)

**Полнота:** есть ли в проекте важные вещи, не отражённые в CLAUDE.md? (новые скрипты в package.json, новые директории, изменившийся стек)

**Размер:** CLAUDE.md больше 50 строк → предложить вынести детали в `.claude/rules/`.

**Разбиение по paths:** если проект многокомпонентный, а файлы в `.claude/rules/` не используют `paths` frontmatter — предложить добавить, чтобы backend-контекст не загружался при работе с фронтендом и наоборот.

Для каждой найденной проблемы:
- Покажи что не так
- Предложи конкретное исправление (diff)
- Объясни одним предложением почему

Не перезаписывай молча. Покажи все изменения и дождись одобрения.

## Фаза 4: Финализация

Покажи пользователю итоговую структуру:
- Что в CLAUDE.md (и сколько строк)
- Какие файлы в `.claude/rules/`, с какими paths
- Что вынесено и почему

Дождись одобрения перед записью.

Перед показом результата проверь:

- В CLAUDE.md нет информации, которую Claude найдёт в коде сам 
- Серверы и URL проверены на доступность (команды выполнены, результат зафиксирован)
- Секреты (пароли, токены) не захардкожены — ссылки на .env 
- Файлы в .claude/rules/ имеют paths если проект многокомпонентный 
- Нет дублирования между CLAUDE.md и файлами в rules 
- Каждый файл ≤ 50 строк

Если какой-то пункт не выполнен — исправь до показа пользователю.
