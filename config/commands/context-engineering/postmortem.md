# /postmortem

Разбери что пошло не так, почему, и обнови конфигурацию проекта чтобы это не повторилось.

## Ограничения

- Кодовую базу не модифицирует (кроме файлов в `.claude/rules/` и CLAUDE.md)
- Результаты анализа сохраняются в `.context/postmortems/`

## Вход

Пользователь описывает что сломалось. Это может быть:
- «Claude удалил файл, который нельзя было трогать»
- «После рефакторинга перестала работать оплата»
- «Агент изменил публичный API и сломал три клиента»
- «Баг в продакшене из-за пропущенного edge case»
- Или просто «вот этот коммит всё сломал»

Если описание недостаточно — уточни через AskUserQuestion.

## Фаза 1: Сбор фактов

Что произошло — конкретно:
- Какие файлы были изменены
- Какое поведение ожидалось
- Какое поведение получено
- Когда обнаружено (сразу / через время / в продакшене)

Если есть коммит — прочитай diff:
```
git show {commit_hash}
git log --oneline -10
```

Если есть ошибка — прочитай лог, stack trace, вывод теста.

Запусти explore-субагента (thoroughness: medium) для контекста:
- Какие компоненты затронуты
- Кто зависит от изменённого кода
- Были ли тесты на этот случай

Сохрани факты в `.context/postmortems/{дата}_{краткое_имя}/facts.md`.

## Фаза 2: Анализ причин

Определи корневую причину. Не симптом, а причину:
- «Удалил файл» — это симптом. Причина: «В CLAUDE.md не указано, что этот файл нельзя трогать»
- «Сломал API» — симптом. Причина: «Не проверил обратную совместимость, нет правила в rules»
- «Пропущен edge case» — симптом. Причина: «Нет тестов на граничные случаи, нет чеклиста в workflow»

Категории причин:
- **Отсутствие ограничения** — в CLAUDE.md или rules нет правила, которое предотвратило бы ошибку
- **Недостаточное исследование** — агент не нашёл зависимости или аналоги перед изменением
- **Пропущенная верификация** — изменение не проверено (нет тестов, не запущен линтер, не проверена обратная совместимость)
- **Устаревшая документация** — CLAUDE.md или rules содержат неактуальную информацию
- **Архитектурная проблема** — код организован так, что ошибку легко допустить

## Фаза 3: Формирование исправлений

Для каждой корневой причины — конкретное действие:

**Если отсутствие ограничения:**
Предложи добавить правило в CLAUDE.md или `.claude/rules/`. Формулировка — конкретная:
- НЕ «будь осторожнее с API»
- А «Файлы в `src/api/public/` определяют публичный API. Не менять сигнатуры существующих эндпоинтов без обсуждения с пользователем. Добавлять новые — можно.»

**Если недостаточное исследование:**
Предложи дополнение к промпту `define-task` или `research` — что именно нужно проверять дополнительно.

**Если пропущенная верификация:**
Предложи hook или дополнение к верификационному шагу. Конкретная команда или проверка.

**Если устаревшая документация:**
Предложи обновление с diff.

**Если архитектурная проблема:**
Зафиксируй как техдолг в `.context/tech_debt.md` с описанием проблемы и предложением решения.

## Фаза 4: Применение

Покажи пользователю:
- Что произошло (одно предложение)
- Корневая причина (одно предложение)
- Предложенные изменения (diff для каждого файла)

Дождись одобрения через AskUserQuestion.

После одобрения — внеси изменения в CLAUDE.md и/или `.claude/rules/`.

Сохрани полный postmortem в `.context/postmortems/{дата}_{краткое_имя}/report.md`:

```markdown
# Postmortem: {краткое описание}

**Дата:** {дата}
**Серьёзность:** {низкая / средняя / высокая / критическая}

## Что произошло
{Факты}

## Корневая причина
{Одно предложение}

## Категория
{Отсутствие ограничения / Недостаточное исследование / Пропущенная верификация / Устаревшая документация / Архитектурная проблема}

## Внесённые изменения
{Что добавлено/изменено в CLAUDE.md, rules, hooks}

## Как предотвращено повторение
{Какое правило или проверка теперь это ловит}
```
