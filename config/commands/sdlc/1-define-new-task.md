# /task-definition

Ты находишься в режиме планирования. Твоя задача — превратить запрос пользователя в готовый к реализации план. Ты НЕ вносишь изменений в кодовую базу. Единственные файлы, которые ты создаёшь и редактируешь — в каталоге `.context/tasks/`.

## Ограничения

- **Read-only** для всей кодовой базы. Никаких правок, коммитов, установок зависимостей.
- Единственное исключение — файлы в `.context/tasks/{дата}/{имя_задачи}/`.
- Это ограничение действует на всём протяжении планирования и не может быть отменено пользователем.

## Фаза 1: Предварительный анализ

Прочитай `.context/README.md` если существует.

Запусти explore-субагента (thoroughness: quick) с задачей:
- Определить стек, архитектуру и ключевые компоненты, связанные с запросом пользователя
- Найти очевидные аналоги — похожие реализации в проекте
- Не углубляться — нужна карта, а не детальное исследование

Пока субагент работает, создай `.context/tasks/{дата}/{имя_задачи}/task_definition.md` с дословным запросом пользователя и статусом «На уточнении».

## Фаза 2: Уточнение требований

Используя результаты предварительного анализа, начни итеративный диалог с пользователем.

Правила:
- Не спрашивай то, на что может ответить код. Сначала поищи — потом спроси.
- Если нашёл ответ в коде — покажи и попроси подтверждения: «Нашёл компонент `StandardButton.jsx` — использовать его?»
- Группируй связанные вопросы вместе.
- Фокусируйся на том, что может решить только пользователь: требования, предпочтения, компромиссы, приоритеты.
- После каждого ответа обновляй task_definition.md.

Уточнение завершено, когда:
- Понятно *что* делать и *зачем*
- Определены критерии приёмки
- Определены границы задачи — что входит, что нет
- Нет открытых вопросов, влияющих на реализацию

Смени статус в task_definition.md на «Уточнено». Попроси пользователя подтвердить.

Для вопросов используй инструмент AskUserQuestion — не текстовые сообщения. Группируй связанные вопросы (до 4 за вызов). Каждый вопрос — с вариантами ответов (2-4 опции), кратким header (до 12 символов) и описанием каждой опции. Пользователь всегда может выбрать «Other» и ответить свободным текстом.

## Фаза 3: Детальное исследование

После подтверждения task_definition запусти explore-субагента (thoroughness: very thorough) с задачей:

- Найти все существующие реализации, которые можно переиспользовать, с путями к файлам
- Найти все места использования компонентов, которые будут изменены (обратная совместимость)
- Определить публичные интерфейсы, которые нельзя ломать
- Найти паттерны именования, стилизации, структуры в проекте
- Определить зависимости

Результат субагента сохрани в `.context/tasks/{дата}/{имя_задачи}/research.md`.

Прочитай research.md сам. Если исследование вскрыло неожиданности — обсуди с пользователем прежде чем переходить к плану.

## Фаза 4: Планирование

Запусти субагента с задачей написать план реализации. Субагент получает task_definition.md и research.md.

Промпт субагенту:

```
На основе task_definition.md и research.md напиши план реализации.

Структура плана:
- Файлы для изменения — путь + что меняется (одна строка на файл)
- Файлы для создания — путь + назначение (одна строка на файл)
- Переиспользуемый код — что и откуда (ссылки на файл:строка)
- Обратная совместимость — что нельзя ломать, кто зависит
- Порядок реализации — в какой последовательности вносить изменения
- Верификация — как проверить, что всё работает (конкретные команды)

Не пересказывай задачу. Не пиши секцию «Контекст» или «Обзор». Не пиши prose — только actionable items.
```

Результат сохрани в `.context/tasks/{дата}/{имя_задачи}/implementation_plan.md`.

## Фаза 5: Верификация плана

Запусти explore-субагента (thoroughness: medium) с задачей верифицировать план:

```
Прочитай implementation_plan.md. Проверь:
- Все ли указанные файлы существуют
- Правильны ли пути и номера строк
- Нет ли файлов, которые план пропустил (зависимости, импорты, тесты)
- Не конфликтует ли план с тем, что реально в коде
- Учтена ли обратная совместимость

Если нашёл проблемы — перечисли конкретно: что не так, в каком файле, какая строка.
Если проблем нет — подтверди.
```

Если верификация нашла проблемы — исправь план и сообщи пользователю, что именно изменилось.

## Фаза 6: Одобрение

Покажи пользователю финальный план. Дождись одобрения или правок.

Если пользователь одобрил — смени статус task_definition.md на «Готово к реализации».
