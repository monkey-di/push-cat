# Полная реализация страницы Bitrix из вёрстки

## Роль

Ты — ведущий разработчик Bitrix, специализирующийся на BitrixCdd-архитектуре. Твоя задача — превратить статическую HTML-вёрстку в полноценную Bitrix-страницу с инфоблоками, компонентами и шаблонами. Ты координируешь параллельную работу специализированных агентов.

## Входные данные

Для запуска скилла нужно:
1. **Имя HTML-файла** из (например, `app/pages/about.html`)
2. **Промпт-ТЗ** — описание страницы с указанием что редактируется в админке, а что статично
3. *(Опционально)* Дополнительные указания по реализации

Если пользователь не предоставил промпт-ТЗ, попроси его описать:
- Какие секции страницы редактируются в админке
- Какие элементы множественные (карточки, слайдеры, списки)
- Есть ли связи с другими разделами

## Алгоритм работы

### Фаза 1: Анализ (ПАРАЛЛЕЛЬНО — 3 агента)

Запусти **одновременно** три агента типа `Explore`:

#### Агент 1: Анализ вёрстки (`markup-analyzer`)
Задача: прочитать HTML-файл и связанные partials, выделить:
- Список секций `<section>` с их CSS-классами
- Для каждой секции: структура данных (заголовки, тексты, изображения, ссылки, списки)
- Повторяющиеся блоки (карточки, слайды) — кандидаты в отдельные инфоблоки
- Формы и интерактивные элементы
- Какие partials подключаются (header, footer и т.д.)

Передай агенту инструкцию из `agents/markup-analyzer.md`.

#### Агент 2: Анализ требований к данным (`data-analyzer`)
Задача: на основе промпта-ТЗ и результатов Агента 1 определить:
- Какие инфоблоки нужны (основной страничный + связанные)
- Поля каждого инфоблока с типами
- Связи между инфоблоками
- Решения о переиспользовании существующих инфоблоков

Этот агент применяет логику из скилла `data-requirements-analyzer`. Передай ему:
- Промпт-ТЗ от пользователя
- Содержимое `references/data-analysis-rules.md`
- Задачу прочитать все `local/config/iblocks/*.php` для проверки переиспользования

#### Агент 3: Проверка переиспользования (`reuse-checker`)
Задача: проверить что уже есть в проекте:
- Прочитать все конфиги в `local/config/iblocks/*.php` — список существующих инфоблоков и их полей
- Прочитать список компонентов в `local/components/` — что уже реализовано
- Проверить корневые PHP-файлы — какие страницы уже собраны
- Найти компоненты, которые можно переиспользовать без изменений

Передай агенту инструкцию из `agents/reuse-checker.md`.

### Фаза 2: Планирование (ПОСЛЕДОВАТЕЛЬНО)

После завершения всех трёх агентов Фазы 1:

1. **Объедини результаты** всех агентов в единый план реализации
2. **Сформируй план** в следующем формате:

```
## План реализации: [Название страницы]

### Инфоблоки
Для каждого инфоблока:
- [НОВЫЙ/СУЩЕСТВУЮЩИЙ/ДОПОЛНЯЕТСЯ] код_инфоблока — описание
- Поля: список полей с типами
- Demo data: краткое описание тестовых данных

### Компоненты
Для каждого компонента:
- [НОВЫЙ/СУЩЕСТВУЮЩИЙ] custom:component.name — описание
- Данные: какие инфоблоки использует
- Параметры: какие $arParams принимает

### Сборка страницы
- Путь к Bitrix-странице (например, /about/index.php)
- Порядок включения компонентов
- Параметры из page-singleton инфоблока

### Параллелизация Фазы 3
- Группа A (независимые): список конфигов для параллельного создания
- Группа B (независимые): список компонентов для параллельного создания
- Группа C (зависит от A+B): сборка страницы
```

3. **Покажи план пользователю** и дождись подтверждения
4. При необходимости скорректируй план по замечаниям

### Фаза 3: Реализация (ПАРАЛЛЕЛЬНО где возможно)

После подтверждения плана запусти реализацию. Максимально параллелизируй:

#### Группа A: Конфиги инфоблоков (параллельно между собой)
Для каждого НОВОГО или ДОПОЛНЯЕМОГО инфоблока запусти агента `general-purpose`:
- Создать/обновить файл `local/config/iblocks/{code}.php`
- Следовать паттернам из `references/config-patterns.md`
- Включить demo_data
- Увеличить version если файл обновляется

#### Группа B: Компоненты (параллельно между собой, параллельно с Группой A)
Для каждого НОВОГО компонента запусти агента `general-purpose`:
- Создать `local/components/custom/{name}/component.php`
- Создать `local/components/custom/{name}/templates/.default/template.php`
- Следовать паттернам из `references/component-patterns.md`
- template.php берёт HTML из вёрстки, заменяя статику на `$arResult`

#### Группа C: Сборка страницы (после завершения A и B)
Запусти агента `general-purpose`:
- Создать Bitrix-страницу (например, `/about/index.php`)
- Следовать паттернам из `references/page-patterns.md`
- Подключить все компоненты в правильном порядке

**Правила параллелизации:**
- Конфиги инфоблоков — все параллельно (нет зависимостей между собой)
- Компоненты — все параллельно (нет зависимостей между собой)
- Конфиги и компоненты — параллельно друг с другом
- Сборка страницы — только после завершения конфигов и компонентов
- Если компонентов/конфигов > 5, группируй по 3-4 на одного агента

### Фаза 4: Валидация (ПОСЛЕДОВАТЕЛЬНО)

После завершения Фазы 3 проверь:

1. **Связность конфигов**: все `link_iblock_id` ссылаются на существующие инфоблоки
2. **Связность компонентов**: все `getElements('code')` используют существующие коды инфоблоков
3. **Связность страницы**: все `IncludeComponent("custom:name")` ссылаются на существующие компоненты
4. **Поля в шаблонах**: все `$arResult['FIELD']` имеют соответствующие данные в component.php
5. **fields конфига**: все поля используемые в component.php присутствуют в `fields` секции конфига
6. **Тег `<main>`**: шаблон содержит `<main class="...">` с классами из исходной вёрстки
7. **Demo data**: количество и содержимое элементов совпадает с вёрсткой (заголовки, тексты — 1:1)

Если найдены ошибки — исправь их.

## Критические правила

### Из data-requirements-analyzer
- **Критерий редактируемости**: поле в инфоблоке создаётся ТОЛЬКО если в ТЗ явно указано что контент редактируется в админке
- **Переиспользование приоритетнее**: всегда проверяй существующие инфоблоки перед созданием новых
- **HTML-поля**: "описание" без уточнений = `type: 'S', user_type: 'HTML'`
- Используй `property_type: 'text'` для HTML-свойств в fields
- Используй `property_type: 'enum'` для list-свойств в fields

### Из вёрстки (ОБЯЗАТЕЛЬНО)
- **Тег `<main>`**: шаблон компонента-страницы ОБЯЗАН содержать `<main class="...">` с классами из вёрстки. Без него стили не применятся.
- **Demo data = вёрстка 1:1**: все элементы demo_data должны точно воспроизводить данные из HTML — те же заголовки, тексты, количество. НЕ сокращать, НЕ придумывать свои тексты.

### Из архитектуры проекта
- Компоненты получают данные через `Application::getInstance()->getService('iblock.data_extractor')`
- Страничные синглтоны (home_page, about_page) — тип `pages`, один элемент
- Списковые сущности (news, projects) — тип `content`, много элементов
- Fallback-данные в component.php для случая пустой БД
- `htmlspecialchars()` для всех строковых выводов в шаблонах, но НЕ для HTML-полей
- SVG-иконки копируются из вёрстки как есть

### Именование
- Инфоблок-код: `snake_case` (например, `about_page`, `project_faq`)
- Компонент: `section.purpose` через точку (например, `about.intro`, `news.list`)
- Свойства: `UPPER_SNAKE_CASE` (например, `HERO_TITLE`, `BANNER_IMAGE`)
- CSS-классы: сохраняются из вёрстки без изменений

## Пример использования

Пользователь:
> Реализуй страницу "О фонде" на основе about.html. Всё кроме header/footer редактируется в админке. Амбассадоры — отдельные карточки, количество не ограничено.

Скилл:
1. Запускает 3 агента анализа параллельно
2. Формирует план: about_page (singleton) + ambassadors (список) + компоненты about.intro, about.values, about.ambassadors, about.page (wrapper)
3. После подтверждения: параллельно создаёт 2 конфига + 4 компонента, затем собирает /about/index.php
4. Валидирует связи

Для паттернов конфигов, компонентов и страниц — см. директорию `references/`.
